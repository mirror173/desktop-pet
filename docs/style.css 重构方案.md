# style.css 重构方案

## 一、背景与目标

当前项目的 renderer 层样式全部集中在单文件 `style.css` 中，随着功能持续增加，样式文件已经出现以下问题：

- 样式体量持续膨胀，阅读和定位成本高
- 全局样式、布局样式、通用组件样式、业务面板样式混杂在一起
- 样式组织方式与现有已模块化的 JS 结构不匹配
- 后续新增功能时，容易在同一文件中继续堆叠，导致耦合加重
- 难以按业务模块快速维护、排查和迭代

本次重构的目标是：

1. 对 `style.css` 做结构化整理与模块化拆分
2. 按基础样式、全局布局、通用组件、业务面板、工具样式进行分层
3. 新建分类目录，通过 `@import` 提供统一聚合入口
4. 仅做归类迁移与格式化整理，不修改样式逻辑，不改变界面表现
5. 保持最小改动原则，确保 UI 完全一致、无样式回归

---

## 二、重构原则

### 1. 纯整理，不改逻辑

本次只做文件归类和代码组织，不做以下操作：

- 不修改任何 CSS 属性值
- 不调整选择器语义
- 不改动层叠优先级
- 不新增或删除样式规则
- 不改动界面表现
- 不引入新的交互或视觉效果

### 2. 最小迁移范围

只移动已有内容到新的分类文件中，保持：

- 原有注释尽量保留
- 原有代码顺序尽量保持
- 原有空行与段落分隔尽量保留
- 原有命名不变

### 3. 入口统一

最终对外只保留一个样式入口文件，由该入口文件通过 `@import` 汇总各分类样式，避免外部依赖多个 CSS 文件。

### 4. 结构先于优化

本次重点在于“可维护性”和“结构清晰”，不追求同步做样式语义重构或命名规范大整顿，以避免一次性变更过大。

---

## 三、推荐拆分方案

采用“轻量拆分”方案，将现有 `style.css` 按功能层次拆分为五类。

### 1. 基础样式

存放最底层、最基础的全局声明，例如：

- `body` 基础设置
- 全局字体、颜色、背景
- 共享的基础动画定义
- 全局通用的状态类

该层是全局样式的起点，应尽量稳定。

### 2. 全局布局

存放页面级布局相关样式，例如：

- 主容器布局
- 窗口级尺寸、边距、圆角、溢出控制
- 页面整体 drag / no-drag 相关布局规则
- 影响整体结构但不属于单个组件的规则

### 3. 通用组件

存放可在多个场景复用的组件样式，例如：

- 按钮
- 卡片
- 输入框
- 弹层
- 滚动条美化
- 通用提示、气泡、标签等

这部分是后续最容易沉淀共用能力的区域。

### 4. 业务面板

存放与具体功能面板相关的样式，例如：

- 文件查找面板
- 打卡面板
- 飞书控制中心
- 会议助手相关面板
- 其他与业务强绑定的视图模块

这一层与 JS 模块的对应关系最强，便于按业务定位和维护。

### 5. 工具样式

存放偏辅助性质的样式，例如：

- 工具类 class
- 显隐控制
- 间距辅助
- 文本截断
- 临时调试辅助类

这部分建议放在最末尾，作为补充层。

---

## 四、建议目录结构

建议在 `renderer` 或 `styles` 相关目录下建立结构化样式目录。由于当前项目以单页 Electron 形态为主，推荐采用如下组织方式：

```text
style/
  index.css
  base/
    reset.css
    variables.css
    animations.css
    typography.css
  layout/
    app.css
    window.css
  components/
    buttons.css
    cards.css
    dialogs.css
    scrollbars.css
  panels/
    pet.css
    file-search.css
    knowledge.css
    lark.css
    briefing.css
  utilities/
    helpers.css
    states.css
```

### 说明

- `index.css` 作为唯一入口，负责 `@import`
- 目录名仅作为分类容器，不要求一次性覆盖全部未来模块
- 初次拆分可先创建“空目录 + 迁移已有内容”，后续再逐步细化
- 如果某些样式暂时没有明确归属，可先放入对应层级中最接近的文件，避免为了追求完美分类引入额外改动

---

## 五、文件拆分建议

### 1. `index.css`

职责：统一聚合入口。

建议仅保留 `@import` 声明，例如按以下顺序：

1. 基础样式
2. 全局布局
3. 通用组件
4. 业务面板
5. 工具样式

这样可以保证层级关系清晰，也方便日后扩展。

### 2. `base` 层

建议迁入内容：

- `body`、全局字体与背景
- `:root` 或全局变量声明（如果已有）
- 统一动画定义
- 全局默认文本规范

### 3. `layout` 层

建议迁入内容：

- `#pet` 等页面根容器
- 影响整个窗口布局的容器样式
- 任何跨组件但只服务于整体页面结构的规则

### 4. `components` 层

建议迁入内容：

- `reminder-overlay`
- `reminder-card`
- `reminder-text`
- 气泡、按钮、弹窗、滚动条等共享组件

### 5. `panels` 层

建议迁入内容：

- 与文件搜索面板直接相关的样式
- 与知识库直接相关的样式
- 与飞书、会议助手、打卡等业务功能直接绑定的样式

### 6. `utilities` 层

建议迁入内容：

- `.no-drag`
- 通用辅助类
- 极少量工具性规则

---

## 六、迁移策略

### 阶段 1：梳理现有样式边界

先对原始 `style.css` 完整扫描并标注区域，按以下方式初步分段：

- 基础全局区
- 主窗口与布局区
- 提示/弹层区
- 面板区
- 工具/辅助区

这一阶段只做归类标记，不改内容。

### 阶段 2：按分类建立新文件

根据分段结果创建对应文件，并将原内容原样迁移。

注意：

- 不要在迁移过程中顺手修正视觉问题
- 不要合并相似但不完全相同的规则
- 不要删除看似冗余但实际上可能被依赖的定义

### 阶段 3：入口文件聚合

通过 `@import` 将各文件统一引入。

建议保持导入顺序稳定，不要随意打乱，以降低层叠行为变化风险。

### 阶段 4：回归验证

迁移完成后进行完整比对，确认：

- 页面外观一致
- 布局无偏移
- 动画无变化
- 按钮、卡片、弹层、面板都无样式回归
- 控制台无 CSS 加载错误

---

## 七、与现有 JS 模块对齐方式

虽然这次采用“轻量拆分”，但样式文件依然建议尽量与现有 JS 功能块保持同一命名思路，便于维护人员快速定位。

建议对应关系如下：

- `renderer` 主壳样式 → `layout`
- 桌宠与提醒相关样式 → `components` 或 `panels/pet.css`
- 文件搜索相关样式 → `panels/file-search.css`
- 知识库相关样式 → `panels/knowledge.css`
- 飞书相关样式 → `panels/lark.css`
- 会议/周报/打卡相关样式 → `panels/briefing.css` 或独立子文件

如果当前某些功能样式比较少，也可以先保留在同一面板文件中，不强行拆得过细。

---

## 八、约束与风险控制

### 必须遵守的约束

- 仅移动，不重写
- 仅整理，不优化视觉
- 仅分层，不改逻辑
- 仅结构调整，不引入新规则

### 主要风险

1. **导入顺序变化导致层叠变化**
   - 处理方式：严格保持原始规则顺序映射

2. **遗漏某些跨区域共享样式**
   - 处理方式：先完整扫描，再迁移；不要基于直觉删改

3. **业务面板归类不清导致后续查找困难**
   - 处理方式：采用“轻量拆分 + 明确文件命名”策略，避免过度抽象

4. **引入路径错误导致样式未加载**
   - 处理方式：统一只保留一个入口文件，并做完整启动验证

---

## 九、验证标准

本次重构完成后，需要满足以下验收条件：

- `style.css` 已拆分为多个结构化文件
- 顶层入口文件通过 `@import` 成功聚合
- 页面 UI 与重构前完全一致
- 无明显样式回归、错位、闪烁、缺失
- 业务面板仍可正常显示与切换
- 控制台无样式加载或路径错误

---

## 十、建议执行顺序

建议按以下顺序推进：

1. 读取并标注现有 `style.css` 全文结构
2. 确认分类边界
3. 创建新目录和入口文件
4. 原样迁移内容到对应分类文件
5. 用 `@import` 聚合
6. 启动项目并验证 UI 完整性
7. 发现问题后仅做必要的路径或归类修正

---

## 十一、后续演进建议

本次完成“轻量拆分”后，后续可以逐步演进为：

- 更细粒度的组件化样式
- 变量集中管理
- 主题色统一抽离
- 面板样式与功能模块进一步一一对应
- 构建阶段自动检查未使用样式或重复样式

但这些都应作为后续迭代，不纳入本次最小改动范围。

---

## 十二、结论

本次 `style.css` 重构以“结构清晰、最小改动、零视觉变化”为核心目标，采用轻量拆分策略最适合当前项目阶段。

最终交付应是：

- 单一入口
- 分层明确
- 归类清晰
- 维护成本更低
- UI 行为完全不变

这能在不影响现有功能的前提下，为后续样式维护和模块化迭代打下基础。
