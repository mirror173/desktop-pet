# `renderer.js` 重构方案
## 1. 背景
当前 `renderer.js` 已经演化成一个超大单文件，里面同时承担了以下职责：
- 飞书 / Lark 登录、发送、轮询、消息解析、UI 状态同步
- 聊天 / 记忆 / 知识库 / LLM 交互
- 计划 / 提醒 / 打卡 / 轨迹等时间管理逻辑
- 文件搜索 / 控制中心 / 高级面板等 UI 面板逻辑
- 角色动画 / 气泡消息 / 拖拽 / 单击双击 / 悬停等交互逻辑
这类结构会带来几个直接问题：
- 入口过长，难以定位逻辑
- UI、数据、通信、状态混在一起，修改容易互相影响
- 某个功能的小改动会牵动整份文件，风险高
- 未来如果继续增加新功能，`renderer.js` 会进一步膨胀
本次重构目标不是改变产品行为，而是把现有逻辑按职责拆开，让 `renderer.js` 回到“入口协调层”的定位。
---
## 2. 重构目标
### 2.1 必须保持不变
- 现有用户可见行为不变
- 飞书 / Lark 功能不变
- 聊天、记忆、知识库、LLM 行为不变
- 计划、提醒、打卡、轨迹行为不变
- 文件搜索、控制中心、高级面板行为不变
- 角色动画、气泡展示、拖拽交互行为不变
### 2.2 重构后希望达成
- `renderer.js` 只负责初始化、事件协调、模块装配
- 每个功能域拥有自己的模块边界
- 公共工具沉到 `src/renderer/utils/`
- UI 事件和业务逻辑分离
- 对主进程 IPC 的调用统一封装，避免页面层直接散落调用
---

结合当前 `src` 结构，可以看到主进程已经开始按职责拆分：
- `src/main/ipc/lark-auth.ipc.js`
- `src/main/ipc/lark-chat.ipc.js`
- `src/main/ipc/lark-work.ipc.js`
- `src/main/ipc/lark-todo.ipc.js`
- `src/main/ipc/file-search.ipc.js`
- `src/main/ipc/knowledge.ipc.js`
- `src/main/ipc/pet.ipc.js`

- `src/renderer/utils/text.js`

说明主进程已经具备“按领域拆服务”的方向，`renderer.js` 也应该同步采用同样的拆分方式。
---
## 4. 建议的拆分边界
### 4.1 飞书 / Lark 相关逻辑
**职责范围**
- 飞书登录状态检查
- 登录 / 切换账号 / 登出
- 飞书消息发送
- 拉取消息并识别回复
- 日程、待办、周报、到岗梳理等工作流
- 飞书配置面板 UI 同步
**建议拆分为**
- `src/renderer/features/lark/index.js`
- `src/renderer/features/lark/auth.js`
- `src/renderer/features/lark/chat.js`
- `src/renderer/features/lark/workflow.js`
- `src/renderer/features/lark/ui.js`
**理由**
这部分逻辑与主进程 `src/main/ipc/lark*.js` 已经天然对应，拆出来后最容易形成稳定边界。
---
### 4.2 聊天 / 记忆 / LLM 相关逻辑
**职责范围**
- 输入解析与意图识别
- 记忆回收、知识库检索、画像更新
- LLM 调用与超时控制
- 文本回复生成
- 上下文记录、回复去重、称呼策略
- 网页助手模式
**建议拆分为**
- `src/renderer/features/chat/index.js`
- `src/renderer/features/chat/memory.js`
- `src/renderer/features/chat/llm.js`
- `src/renderer/features/chat/knowledge.js`
- `src/renderer/features/chat/reply-policy.js`
- `src/renderer/features/web-assistant/index.js`
**理由**
这部分是目前 `renderer.js` 中最重的业务核心之一，且内部纯逻辑较多，适合先拆出可测试的函数模块。
---
### 4.3 计划 / 提醒 / 打卡相关逻辑
**职责范围**
- 计划卡片
- 提醒触发与后续动作
- 周期任务
- 打卡列表与倒计时
- 轨迹记录
- 工作复盘生成
**建议拆分为**
- `src/renderer/features/planner/index.js`
- `src/renderer/features/planner/reminder.js`
- `src/renderer/features/checkin/index.js`
- `src/renderer/features/trajectory/index.js`
- `src/renderer/features/work-review/index.js`
**理由**
这类逻辑大多围绕本地状态和定时器展开，拆开后可以把“状态管理”和“视图渲染”分层处理。
---
### 4.4 文件搜索 / 控制中心 / 高级面板相关逻辑
**职责范围**
- 文件搜索结果查询、分组、排序、展开收起
- 控制中心快捷链接管理
- 高级面板开关、折叠、自动收起
- 面板按钮图标编辑
- 面板拖拽与窗口模式切换
**建议拆分为**
- `src/renderer/features/file-search/index.js`
- `src/renderer/features/control-center/index.js`
- `src/renderer/features/advanced-menu/index.js`
- `src/renderer/features/panels/window-drag.js`
**理由**
这些功能都属于“面板型 UI”，可以统一抽象为可开关、可渲染、可绑定事件的模块。
---
### 4.5 角色动画 / 气泡 / 交互效果相关逻辑
**职责范围**
- 角色头像切换
- 双击动效、拖拽动效、空闲行为
- 气泡展示、自动隐藏、拖动窗口
- 单击、长按、右键、离开、松手等交互
- 角色状态机
**建议拆分为**
- `src/renderer/features/avatar/index.js`
- `src/renderer/features/bubbles/index.js`
- `src/renderer/features/character-input/index.js`
- `src/renderer/features/idle-behavior/index.js`
**理由**
这部分主要是表现层和交互层，和聊天业务相对独立，拆出来后更容易调试动画问题。
---
## 5. 推荐目录结构
建议将 `renderer.js` 逐步拆为下面结构：
```text
src/
  renderer/
    app/
      bootstrap.js
      state.js
      bindings.js
    features/
      lark/
      chat/
      planner/
      checkin/
      trajectory/
      file-search/
      control-center/
      advanced-menu/
      avatar/
      bubbles/
      web-assistant/
    utils/
      text.js
      time.js
      storage.js
      dom.js
      ipc.js
```
### 职责约定
- `app/`：启动、组合、统一注册事件
- `features/`：按业务域拆分的功能实现
- `utils/`：可复用工具函数，不直接依赖具体业务
- `renderer.js`：保留入口文件，只做少量 bootstrap
---
## 6. 迁移策略
### 第一阶段：只拆不改行为
先把大文件里的纯函数、状态计算、格式化逻辑抽出，不改现有页面行为。
优先拆这些：
- 文本处理
- 时间格式化
- 记忆与知识库索引
- 回复策略
- 计划时间解析
### 第二阶段：按功能域搬运事件绑定
把各面板的 DOM 绑定迁移到各自模块中，入口文件只负责调用初始化函数。
### 第三阶段：收口公共能力
把以下能力统一封装：
- `window.api` 调用
- localStorage 读写
- 气泡显示与隐藏
- 角色状态切换
- 面板切换与窗口模式切换
### 第四阶段：清理旧代码
当模块稳定后，删除 `renderer.js` 中已经迁出的逻辑，避免重复实现。
---
## 7. 模块接口约定
为了避免再次变回“大泥球”，建议每个模块都遵守统一形式：
- `init(deps)`：初始化并绑定事件
- `destroy()`：可选，清理监听器和定时器
- `state`：模块内部状态不直接暴露，必要时只暴露读取方法
- `actions`：对外提供有限动作接口
### 依赖注入建议
模块不要直接读取全局变量，统一通过参数传入：
- `window.api`
- `document`
- `localStorage`
- `showMessage`
- `setInputMode`
- `expandPlannerPanel`
- `collapseAdvancedMenu`
这样可降低模块耦合，也更方便未来测试。
---
## 8. 风险与约束
### 8.1 风险
- `renderer.js` 逻辑太多，拆分时容易漏掉隐性依赖
- 某些 UI 逻辑与状态强耦合，迁移后容易出现时序问题
- 大量定时器、事件监听器需要小心释放
### 8.2 约束
- 不改现有业务结果
- 不改页面布局和视觉表现
- 不引入额外的框架迁移
- 不在本轮同时做大规模 UI 重设计
---
## 9. 推荐实施顺序
1. 先拆 `src/renderer/utils/` 下的纯函数
2. 再拆聊天 / 记忆 / LLM 逻辑
3. 再拆飞书 / Lark 逻辑
4. 然后拆计划 / 提醒 / 打卡
5. 最后拆文件搜索、控制中心、高级面板、角色动画
如果目标是尽快降低 `renderer.js` 复杂度，我建议第一批优先做：
- `聊天 / 记忆 / LLM`
- `飞书 / Lark`
因为这两块依赖边界最清晰，迁移收益最大。
---
## 10. 成功标准
当本次重构完成时，应该满足：
- `renderer.js` 文件明显变短，只保留入口和协调代码
- 每个功能域都能在独立文件中找到主逻辑
- 公共工具不再散落在业务文件里
- 代码搜索时可以快速定位“飞书”“聊天”“提醒”“动画”等职责
- 后续新增功能时，不必继续往 `renderer.js` 里堆
---
## 11. 结论
这次重构的核心不是“拆文件”本身，而是把 `renderer.js` 从“所有事情都做”改成“只负责调度”。
推荐采用“按业务域拆分 + 公共工具收口 + 入口薄化”的方案，并优先迁移：
1. 聊天 / 记忆 / LLM
2. 飞书 / Lark
3. 计划 / 提醒 / 打卡
这样能在不改变行为的前提下，最有效地降低维护成本。
